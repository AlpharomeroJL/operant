# Spec: backends

One trait: complete(request) -> stream, where request carries role (planner|grounder), messages, optional images, optional tool schemas. One OpenAI-compatible HTTP client parameterized by a quirk table per provider: base URL, auth header shape (Bearer, x-api-key, query param), payload dialect (openai|anthropic|gemini), streaming frame format, vision attachment encoding, max-token field name. Providers shipped in the table: Ollama, llama.cpp server, LM Studio, vLLM, generic (user base URL plus optional key), OpenAI, Anthropic, Google Gemini, DeepSeek, MiniMax, Moonshot Kimi, Qwen DashScope, Groq, Mistral, xAI, OpenRouter. Anthropic and Gemini get native dialect implementations; everything else rides the OpenAI dialect. Capability probe on configure: one cheap request per capability (text, vision with a 1x1 png, tool call with a trivial schema) plus a context-length lookup table with a conservative default; store the profile; role assignment validates against it and explains mismatches in plain language ("This model cannot see images, so it cannot find things on screen. Use it for planning, or pick a vision model."). OAuth broker (X16): PKCE with S256, loopback redirect on 127.0.0.1 with an ephemeral port, state and nonce verified, tokens ONLY in the OS credential vault (Windows Credential Manager), never in config or logs; refresh 5 minutes before expiry; refresh failure emits a doctor finding with a one-click re-auth card. Providers: ChatGPT/Codex plan and Claude plan; the quirk table maps each OAuth identity to its API dialect and endpoints, and endpoints are config-table driven so provider-side changes are a data edit, not a code change. Secrets redaction middleware strips keys and tokens from every log line and error (grep-tested against seeded fakes).

# Roadmap: live scheduler (making triggers real)

Status: NOT built. This document specifies what the core needs before the
shell's Schedule action and the dashboard's Up next can do anything real. Lane
B13 wired the shell honestly against the frozen IPC contract; it did NOT make
scheduling work, and it must not be recorded as having done so. Everything below
is the follow-up this lane's live behavior is gated on.

## Where things stand today (the honest baseline)

- `contracts/ipc.md` section 5e reserves exactly two scheduler commands,
  `list_triggers` and `upsert_trigger`, and section 5g lists both among the five
  commands that are "reserved in this contract but have no core wiring yet." The
  contract is explicit: a build that has not wired them MUST answer with error
  code `not_implemented`.
- `crates/scheduler` has the trigger TYPES (`CronTrigger`, `FileWatchTrigger`,
  `WindowAppearsTrigger`, `EmailArrivesTrigger` in `src/trigger.rs`), a pure
  `enqueue` with the unattended replay-only guard (`src/lib.rs`), and an
  in-memory `RunQueue` (`src/queue.rs`). It has NO persistence, NO CRUD, NO
  daemon that owns triggers, computes next-fire, fires them, or forwards events.
  The trigger structs are constructed and dropped inside unit tests; nothing
  stores or lists them.
- `docs/specs/ipc-bridge.md` (the A5 survey) records the gap directly: the
  `library.schedule` action "needs a new command+topic," and `upcoming_schedule`
  is sourced from a fixture because "nothing in the wire protocol models what's
  scheduled next yet."
- The bus already reserves the runtime scheduler topics (`contracts/bus_events.md`
  section "Scheduler"): `trigger.fired`, `schedule.enqueued`, and
  `schedule.rejected` with reason `mode_not_replay` / `scope_conflict`. These
  describe a trigger that already fired or a run that was enqueued or refused.
  They do NOT model "the set of configured triggers" or "the next fire time,"
  which is why the two commands above exist and why the store below is needed.

What the shell does with that baseline (lane B13):

- The Schedule button on a library card calls `upsert_trigger`
  (`ui/src/library/state.ts`), receives `not_implemented`, and surfaces
  "Scheduling isn't available yet ..." (`ui/src/library/strings.ts`). It never
  creates or fakes a trigger and never publishes to the bus.
- The dashboard's Up next calls `list_triggers` (`ui/src/dashboard/state.ts`),
  receives `not_implemented`, and shows "Scheduling isn't available yet." rather
  than fabricated upcoming runs. The old fabricated fixture (`UP_NEXT_FIXTURE`)
  is deleted.
- The command seam is `ui/src/scheduler/commands.ts`
  (`createUnavailableSchedulerCommands`), the request/response sibling of the
  `BusClient` event seam. Swapping it for a sidecar-backed implementation is the
  only shell change needed once the core work below lands.

## What the core must build

### 1. A persistent trigger store

There is no store today. The core needs a durable, single-writer store of
trigger definitions that survives a restart, keyed by a stable `trigger_id`.

- Record shape (matches `list_triggers`' result row, `contracts/ipc.md` 5e):
  `{ trigger_id, kind, workflow_name, spec, enabled }`, where `kind` is one of
  `cron` / `file` / `window` / `email` and `spec` is the kind-specific
  specification the existing `crates/scheduler/src/trigger.rs` structs already
  parse (a cron expression, a directory + glob, a process + title regex, an
  email filter). The trigger structs are already `Serialize`/`Deserialize`, so
  the stored form can be their serialized `spec` plus the envelope fields.
- Operations: create, read-all (for `list_triggers`), update, delete, and
  enable/disable. `upsert_trigger` with no `trigger_id` creates and returns a
  freshly minted id; with a `trigger_id` it updates in place.
- Placement: alongside the recorder database under the canonical data dir
  (`docs/specs/ipc-bridge.md` section 5, `%APPDATA%\dev.operant.shell`), so it
  shares the app's existing single-user storage and backup/export path.
- Determinism: ids are content-free and generated by the core, never by the
  shell; the store is the single source of truth. Tests use a fixed id source
  so a stored-then-listed round trip is byte-stable.

### 2. Wire `list_triggers` and `upsert_trigger` in the serve loop

The `--serve` request/response loop (the shell-to-core bridge, `contracts/ipc.md`
sections 2-5) must stop answering these two commands with `not_implemented` and
instead route them to the store:

- `list_triggers` -> read all records -> return the rows.
- `upsert_trigger` -> validate `spec` against `kind` (reuse the existing trigger
  parsers; `CronTrigger::next_occurrence` already returns `None` for an invalid
  expression, so a bad spec is an `invalid_args` refusal, not a silent accept)
  -> create or update -> return `{ trigger_id }`.

Per the contract's versioning rules (`contracts/ipc.md` section 9), a command
moving from `not_implemented` to implemented is purely additive and does not
bump the protocol version. No shell change is required beyond swapping the
command seam implementation.

Consider adding a next-fire timestamp to the `list_triggers` row (an OPTIONAL
field, additive per section 9) so the dashboard's Up next can show a friendly
"tomorrow at 9 am" label. Today the shell shows the raw `spec` and never
computes fire times itself, because next-occurrence is the core's job, not the
view's.

### 3. A scheduler daemon that fires triggers into runs

The store holds definitions; something must act on them. The core needs a
long-lived scheduler that, from the enabled triggers:

- computes each trigger's next occurrence (cron via the DST-safe
  next-occurrence logic in `docs/specs/scheduler.md`; file/window/email via
  their poll or event sources),
- honors the missed-trigger rules on wake (cron fires once within a 15-minute
  grace, file events are re-scanned, everything else drops with a log line),
- enqueues the matched workflow through `operant_scheduler::enqueue` into the
  serialized `RunQueue`, carrying any trigger-supplied input (for example the
  matched file path), and
- publishes `trigger.fired` then `schedule.enqueued` (or `schedule.rejected`) on
  the bus so the shell's existing handlers render the activity with no new UI.

### 4. Compiled-only replay for unattended runs (the hard safety line)

This is the campaign's compiled-only enforcement for unattended runs, and it is
the reason a live scheduler is gated behind a follow-up rather than shipped now.
An unattended run has no human watching, so it must be deterministic, offline,
and model-free:

- Replay mode only. `operant_scheduler::enqueue` already refuses any run whose
  mode is not `Replay` (`SchedulerError::UnattendedNonReplay`, mirrored on the
  bus as `schedule.rejected` reason `mode_not_replay`), and the refusal is
  typed and tested. Keep that guard at the actual launch path, not only in the
  pure helper: the daemon must never hand a non-replay run to the queue.
- Compiled workflow only. The trigger's `workflow_name` must resolve to a
  COMPILED, installed workflow manifest, never a raw recorded trajectory and
  never a teach/explore goal. The scheduler launches
  `operant_replay::Replayer::replay_compiled` (`crates/replay/src/lib.rs`),
  which depends on `operant-replay` alone and never on
  `operant-orchestrator`/`reqwest` (`docs/specs/ipc-bridge.md` sections 0 and 3b).
  This keeps unattended execution provably free of a model call or a network
  hop: no planner is assembled, so there is nothing to reach out with.
- Fail closed. A trigger that points at a workflow which is not compiled, or is
  missing, is refused (a typed error surfaced as `schedule.rejected`), not
  silently upgraded to a teach run.

## Acceptance for "scheduler is live"

1. `upsert_trigger` persists a trigger that survives a core restart; a following
   `list_triggers` returns it. Round trips are deterministic.
2. Invalid specs are refused with `invalid_args`; non-replay or non-compiled
   targets are refused with the typed `schedule.rejected` reasons, under test.
3. A fired trigger enqueues a compiled replay run and emits `trigger.fired` +
   `schedule.enqueued`; the shell's Up next and Recent runs reflect it with no
   further UI work.
4. The shell swaps `createUnavailableSchedulerCommands` for the sidecar-backed
   commands; the library Schedule action and dashboard Up next light up with no
   other change, and the honest not-implemented path (`ui/src/scheduler/`,
   `ui/src/library`, `ui/src/dashboard` tests) is replaced by real-path tests.
5. The default `just verify` build stays offline and deterministic: unattended
   runs remain compiled-replay only, so the golden and airgap gates stay green.

Until all five hold, scheduling is not available, and the shell says exactly
that.
